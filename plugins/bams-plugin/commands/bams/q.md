# Bams Q

코드베이스에 대한 질문에 답변합니다. 질문에서 범위를 자동 감지하여 관련 코드를 수집한 뒤 프로젝트 컨텍스트에 기반한 답변을 제공합니다.

서브에이전트를 사용하지 않고, 직접 Read/Grep/Glob 도구로 코드를 탐색합니다.

입력: $ARGUMENTS

$ARGUMENTS가 비어있으면, 사용자에게 무엇이 궁금한지 물어보고 중단합니다.

## Phase 1: 사전 조건

Glob으로 `.crew/config.md`가 존재하는지 확인합니다. 있으면 읽어서 프로젝트 컨텍스트를 로드합니다. 없어도 진행 가능합니다 (프로젝트 컨텍스트 없이 답변).

## Phase 2: 범위 자동 감지

$ARGUMENTS를 분석하여 질문의 범위를 자동으로 판별합니다:

### 판별 규칙

**카테고리 A -- 파일 기반**: $ARGUMENTS에 파일 경로 패턴이 포함된 경우
- 슬래시(`/`)가 포함된 경로 (예: `src/auth/middleware.ts`)
- 확장자가 포함된 파일명 (예: `handler.go`)
- -> Phase 3A로 진행

**카테고리 B -- 함수/메서드 기반**: $ARGUMENTS에 함수나 메서드 식별자가 포함된 경우
- camelCase, snake_case, PascalCase 패턴의 식별자
- 괄호가 붙은 호출 형태 (예: `processPayment()`)
- -> Phase 3B로 진행

**카테고리 C -- 기능 설명 기반**: 위 패턴에 해당하지 않는 자연어 설명
- -> Phase 3C로 진행

**복합 케이스**: 가장 구체적인 카테고리 우선 (A > B > C)

## Phase 3A: 파일 기반 코드 수집

1. Glob으로 해당 파일이 존재하는지 확인
   - **존재하면**: Read로 파일 내용을 읽기
   - **존재하지 않으면**: Glob으로 유사한 경로 검색, 후보 제시
2. 파일이 500줄 초과 시: $ARGUMENTS 키워드와 관련된 섹션만 추출

-> Phase 4로 진행

## Phase 3B: 함수/메서드 기반 코드 수집

1. Grep으로 식별자의 **정의(definition)** 위치를 검색 (함수 선언, 클래스 선언)
2. 정의가 포함된 파일을 Read로 읽어 해당 함수/메서드의 전체 코드 추출
3. 같은 파일 내의 관련 타입, 인터페이스, 상수도 포함
4. Grep으로 이 함수/메서드의 **호출자(caller)** 를 최대 5개 검색
5. 호출자의 관련 코드 스니펫 수집

-> Phase 4로 진행

## Phase 3C: 기능 설명 기반 코드 수집

1. 기능 설명에서 핵심 키워드 추출
2. Grep과 Glob으로 관련 파일 탐색 (파일명, 디렉토리명, 코드 내 키워드)
3. 발견된 파일 중 가장 관련성 높은 파일 최대 5개를 Read로 읽기
4. 파일 간의 관계 (import/호출 체인) 추적
5. 해당 기능의 데이터 흐름과 핵심 로직 파악

-> Phase 4로 진행

## Phase 4: 답변

수집된 코드와 프로젝트 컨텍스트를 기반으로 질문에 답변합니다.

답변 방침:
1. **코드 근거 기반**: 추측하지 않고 수집된 코드를 근거로 답변. 코드 참조 시 file:line 형태로 출처를 명시.
2. **구조화된 답변**: 질문 유형에 따라 적절한 구조 사용
3. **프로젝트 맥락 반영**: 프로젝트의 아키텍처와 컨벤션을 고려
4. **간결성**: 핵심을 먼저 말하고 상세는 뒤에
5. **불확실성 표시**: 판단이 어려운 부분은 "추정" 또는 "확인 필요"로 표시

```
## 답변
[핵심 답변]

### 상세
[코드 참조를 포함한 상세 설명]

### 관련 코드
[file:line 형태의 참조 목록]
```
