# Crew Debug

Crew 오케스트레이터로서 버그 조사 → 원인 분석 → 영향 분석 → 수정 → 검증 파이프라인을 실행합니다.

입력: $ARGUMENTS

$ARGUMENTS가 비어있으면, 사용자에게 어떤 버그를 조사할지 물어보고 중단합니다.

## 코드 최신화

Bash로 `git rev-parse --is-inside-work-tree 2>/dev/null`를 실행하여 git 저장소인지 확인합니다.

**git 저장소인 경우**: Bash로 `git branch --show-current`를 실행하여 현재 브랜치를 확인한 뒤, `git pull origin {현재 브랜치}`를 실행하여 원격 저장소의 최신 코드를 가져옵니다. 충돌이 발생하면 사용자에게 알리고 중단합니다.

**git 저장소가 아닌 경우**: 이 단계를 스킵합니다.

## 사전 조건

Glob으로 `.crew/config.md`가 존재하는지 확인합니다. 없으면:
- 출력: "Crew가 초기화되지 않았습니다. `/crew:init`을 실행하여 설정하세요."
- 여기서 중단.

`.crew/config.md`와 `.crew/board.md`를 읽습니다.

## Phase 1: 버그 접수 및 재현

### 1a. 버그 정보 파싱

$ARGUMENTS에서 다음을 추출합니다:
- **증상**: 무엇이 잘못되었는가?
- **에러 메시지**: 스택 트레이스, 에러 로그 (있으면)
- **재현 조건**: 어떤 상황에서 발생하는가? (있으면)

정보가 불충분하면 **AskUserQuestion**으로 추가 정보를 요청:

Question: "버그에 대해 추가 정보가 필요합니다."
Header: "Bug Info"
Options:
- **에러 메시지 있음** - "스택 트레이스나 에러 로그를 붙여넣기"
- **재현 방법 있음** - "재현 단계를 설명"
- **증상만 있음** - "현재 정보로 조사 시작"

### 1b. Reproducer 에이전트

서브에이전트 실행 (Task tool, subagent_type: **"common-plugin:reproducer"**, model: **"sonnet"**):

> **버그 분석 모드**로 보고된 버그를 분석하고 재현을 시도합니다.
>
> **버그 리포트**: [$ARGUMENTS + 추가 정보 삽입]
> **프로젝트 컨텍스트**: [.crew/config.md 내용 삽입]

Reproducer 반환 후, 결과를 사용자에게 간략히 표시:
```
버그 조사 시작
──────────────────────────────────────
증상: [요약]
관련 파일: [N]개 식별
초기 가설: [N]개
테스트 커버리지: [있음/없음]
```

## Phase 1.5: Git 체크포인트

수정을 시작하기 전에 현재 코드 상태를 보존합니다.

먼저 Bash로 `git rev-parse --is-inside-work-tree`를 실행하여 git 저장소인지 확인합니다.

**git 저장소가 아닌 경우**: 이 단계를 스킵하고 Phase 2로 진행합니다.

**git 저장소인 경우**: Bash로 `git status --porcelain`을 실행하여 미커밋 변경사항이 있는지 확인합니다.

**AskUserQuestion**으로 체크포인트 방식을 선택받습니다:

Question: "수정 시작 전 코드를 어떻게 보존할까요?"
Header: "Git"
Options:
- **Feature branch** - "새 브랜치를 생성하여 작업 (예: crew/fix-[slug])"
- **Stash** - "현재 변경사항을 stash하고 현재 브랜치에서 작업"
- **스킵** - "체크포인트 없이 바로 진행"

**Feature branch** 선택 시:
1. 미커밋 변경사항이 있으면 먼저 `git stash`
2. `git checkout -b crew/fix-[slug]` 실행
3. stash가 있으면 `git stash pop`
4. 사용자에게 알림: "브랜치 `crew/fix-[slug]` 생성됨."

**Stash** 선택 시:
1. `git stash push -m "crew: checkpoint before fix-[slug]"` 실행
2. 사용자에게 알림: "체크포인트 생성됨. 되돌리려면: `git stash pop`"

## Phase 2: 원인 분석 (Root Cause Analysis)

**3개 Investigator 에이전트를 병렬로 실행** (Task tool, subagent_type: **"common-plugin:investigator"**, model: **"sonnet"**):

### Investigator 1 — 코드 흐름 추적:

> **관점: 코드 흐름 추적** — 버그의 실행 흐름을 추적합니다.
>
> **버그 정보**: [Phase 1 결과: 관련 파일, 재현 단계, 초기 가설]
> **관련 파일**: [Reproducer가 식별한 파일 목록]

### Investigator 2 — 변경 이력 분석:

> **관점: 변경 이력 분석** — 최근 변경 이력에서 버그 원인을 찾습니다.
>
> **버그 정보**: [Phase 1 결과]
> **관련 파일**: [목록]

### Investigator 3 — 엣지 케이스 및 환경 분석:

> **관점: 엣지 케이스 및 환경 분석** — 엣지 케이스와 환경 요인을 조사합니다.
>
> **버그 정보**: [Phase 1 결과]
> **관련 파일**: [목록]

### 원인 종합

3개 Investigator가 모두 반환한 후:

1. 각 에이전트의 근본 원인 후보를 수집
2. 일치하는 후보가 있으면 신뢰도 높음으로 태깅
3. **근본 원인을 하나로 확정** (여러 후보가 있으면 증거가 가장 강한 것 선택)
4. 확정이 어려우면 사용자에게 후보를 제시하고 선택 요청

사용자에게 표시:
```
원인 분석 완료
──────────────────────────────────────
근본 원인: [한 줄 요약]
위치: [file:line]
유형: [로직 오류 / 엣지 케이스 / 레이스 컨디션 / 회귀 / 환경 / 타입 불일치]
신뢰도: [높음/중간/낮음]
증거: [일치한 Investigator 수]/3
```

## Phase 3: 영향 분석 (Impact Analysis)

수정 전에 사이드이펙트를 사전 분석합니다.

### 3a. Impact Analyzer 에이전트

서브에이전트 실행 (Task tool, subagent_type: **"common-plugin:impact-analyzer"**, model: **"sonnet"**):

> **영향 분석 모드**로 버그 수정의 영향 범위를 분석합니다.
>
> **근본 원인**: [Phase 2에서 확정된 원인]
> **수정 대상 파일**: [file:line]
> **프로젝트 컨텍스트**: [.crew/config.md 내용]

### 3b. 범위 잠금 (Scope Lock)

Impact Analyzer 결과를 기반으로 **수정 허용 파일 목록**을 확정합니다:

1. Impact Analyzer가 권장한 수정 범위를 사용자에게 표시:

```
영향 분석 완료
──────────────────────────────────────
위험 등급: [Low/Medium/High]
직접 영향: [N]개 파일
간접 영향: [N]개 파일
버그 의존 코드: [있음 N개 / 없음]

수정 허용 범위:
  ✎ [file1:line] — [수정 이유]
  ✎ [file2:line] — [수정 이유]
  ⚠ [file3:line] — 버그 의존 코드, 함께 수정 필요

관련 테스트: [N]개
```

2. **버그 의존 코드**가 발견되면 경고:
   ```
   ⚠ 주의: 현재 (잘못된) 동작에 의존하는 코드가 [N]개 있습니다.
   이 코드도 함께 수정하지 않으면 버그 수정 후 새로운 문제가 발생합니다.
   ```

3. **위험 등급이 High**이면 사용자에게 확인:

   **AskUserQuestion**:
   Question: "수정 영향 범위가 넓습니다. 계속 진행할까요?"
   Header: "Risk"
   Options:
   - **진행** - "분석된 범위로 수정 진행"
   - **축소** - "핵심 파일만 수정 (버그 의존 코드는 나중에)"
   - **중단** - "수동으로 처리"

이 단계에서 확정된 **수정 허용 파일 목록**을 Phase 4에서 Developer 에이전트에게 전달합니다.

## Phase 4: 수정

### 4a. Developer 에이전트 — 외과적 패치

서브에이전트 실행 (Task tool, subagent_type: **"common-plugin:developer"**, model: **"opus"**):

> **버그 수정 모드**로 버그를 최소 범위로 수정합니다.
>
> **근본 원인**: [Phase 2 결과]
> **수정 허용 파일**: [Phase 3에서 확정된 목록 — 이 파일만 수정 가능]
> **버그 의존 코드**: [함께 수정해야 할 코드 목록]
> **영향 분석**: [직접/간접 영향 요약]
> **코드베이스 컨벤션**: [CLAUDE.md와 .crew/config.md에서]

### 4b. 범위 위반 검증

Developer 에이전트 반환 후, 수정된 파일 목록을 Phase 3의 수정 허용 파일 목록과 대조합니다.

**허용 범위를 벗어난 파일이 있으면:**
- 사용자에게 경고:
  ```
  ⚠ 범위 위반 감지
  허용 범위 외 수정된 파일:
    - [file] — [변경 내용]
  ```
- **AskUserQuestion**:
  Question: "수정 범위를 벗어난 변경이 있습니다. 어떻게 할까요?"
  Header: "Scope"
  Options:
  - **허용** - "이 파일도 수정 범위에 포함"
  - **되돌리기** - "범위 외 변경만 되돌리기"
  - **전체 되돌리기** - "모든 수정을 되돌리고 재시도"

## Phase 4.5: 근본 원인 검증 (Root Cause Verification)

수정이 증상만 가리는 게 아닌지 검증합니다.

서브에이전트 실행 (Task tool, subagent_type: **"common-plugin:pm-analyst"**, model: **"opus"**):

> **근본 원인 검증 모드**로 버그 수정의 적절성을 검증합니다.
>
> **원래 버그**: [$ARGUMENTS]
> **근본 원인**: [Phase 2 결과]
> **수정 내용**: [Phase 4에서 Developer가 반환한 변경 요약]
> **수정된 파일**: [수정된 파일 목록 — Read로 읽어서 확인]
> **영향 분석**: [Phase 3 결과]

### 검증 결과 처리

**PASS**: Phase 5로 진행.

**WARN**: 사용자에게 경고를 표시하고 진행 여부 확인:
- **AskUserQuestion**:
  Question: "수정에 주의사항이 있습니다. 계속할까요?"
  Header: "Verify"
  Options:
  - **진행** - "경고를 인지하고 계속"
  - **보완** - "추가 수정 사항 적용 후 진행"
  - **중단** - "수정을 되돌리고 수동 처리"

**FAIL**:
- 사용자에게 실패 내용 표시
- Developer 에이전트 재실행 (검증 피드백 포함, Phase 4a로 돌아감)
- **최대 2회 재시도**, 초과 시 사용자에게 수동 처리 안내

## Phase 5: 변경사항 확인

git 저장소인 경우, 수정 내용을 사용자에게 보여주고 확인받습니다.

1. Bash로 `git diff --stat` 실행하여 변경 요약 표시
2. Bash로 `git diff` 실행하여 상세 diff 표시

사용자에게 표시:
```
버그 수정 완료 — 변경사항 확인
──────────────────────────────────────
[git diff --stat 출력]
```

**AskUserQuestion**으로 확인:

Question: "수정 결과를 적용할까요?"
Header: "Confirm"
Options:
- **적용** - "변경사항을 유지하고 다음 단계로 진행"
- **되돌리기** - "모든 변경사항을 되돌리고 중단"
- **부분 되돌리기** - "특정 파일만 되돌리기"

**적용** 선택 시: Phase 6으로 진행.

**되돌리기** 선택 시:
- `git checkout -- .` 실행
- 새로 생성된 파일은 수동으로 삭제 확인
- "변경사항이 되돌려졌습니다." 메시지 출력 후 중단

**부분 되돌리기** 선택 시:
- 변경된 파일 목록을 보여주고 되돌릴 파일을 AskUserQuestion (multiSelect)으로 선택
- 선택된 파일에 대해 `git checkout -- [file]` 실행
- Phase 6으로 진행

## Phase 6: 회귀 테스트

### 6a. 기존 테스트 실행

프로젝트에 테스트 러너가 있으면 (package.json scripts, Makefile, go test 등):

1. Bash로 기존 테스트 스위트 전체 실행
2. 결과 캡처

**테스트 실패 시:**
- 실패한 테스트 목록을 사용자에게 표시
- **AskUserQuestion**:
  Question: "기존 테스트가 실패했습니다. 어떻게 할까요?"
  Header: "Tests"
  Options:
  - **수정** - "실패한 테스트가 버그 수정을 반영하도록 업데이트"
  - **조사** - "테스트 실패가 회귀인지 확인"
  - **스킵** - "테스트 실패를 무시하고 진행"

  **수정** 선택 시: 실패한 테스트가 버그의 (잘못된) 이전 동작을 테스트하는 경우 → 테스트를 올바른 동작으로 업데이트
  **조사** 선택 시: 실패 원인을 분석하여 회귀인지 의도된 변경인지 판단 → 결과에 따라 처리

### 6b. 회귀 테스트 생성

QA 에이전트 실행 (Task tool, subagent_type: **"common-plugin:qa-engineer"**, model: **"sonnet"**):

> **회귀 테스트 모드**로 버그 수정에 대한 회귀 테스트를 작성합니다.
>
> **원래 버그**: [$ARGUMENTS — 증상 설명]
> **근본 원인**: [Phase 2 결과]
> **수정된 파일**: [Phase 4에서 수정된 파일 목록]
> **수정 내용**: [변경 요약]
> **테스트 위치**: [.crew/config.md의 test_dir 설정]
> **기존 테스트 패턴**: [프로젝트의 테스트 프레임워크와 패턴]

QA 에이전트 반환 후:
- 생성된 테스트를 실행하여 통과 여부 확인
- 실패하면 수정 (최대 2회 재시도)

## Phase 7: 마무리

### 7a. README.md 동기화

프로젝트 루트에 `README.md`가 존재하는지 Glob으로 확인합니다.

**`README.md`가 존재하면:**

이번 버그 수정에서 변경된 파일 목록을 기반으로, README.md에 영향을 줄 수 있는 변경이 있는지 판단합니다:
- API 동작 변경 (파라미터, 응답, 에러 코드 등)
- 설정/환경변수 변경
- 사용법 또는 제약사항 변경
- 알려진 이슈/제한사항 업데이트

**영향이 있으면** `README.md`를 Read로 읽고, 변경사항에 맞춰 해당 섹션을 Edit으로 업데이트합니다. 기존 README.md의 스타일과 구조를 유지합니다.

**`README.md`가 없거나 영향이 없으면:** 이 단계를 스킵합니다.

### 7b. 아티팩트 저장

버그명에서 slug 생성 (예: "null-pointer-auth" 등).

디버그 리포트를 `.crew/artifacts/review/debug-[slug].md`에 저장합니다. 다음 섹션을 포함:
- **근본 원인**: 유형, 위치(file:line), 설명, 도입 시점
- **영향 분석**: 위험 등급, 직접 영향 파일 수, 버그 의존 코드 유무
- **수정 내용**: 수정 파일, 변경 요약, 근본 원인 검증 결과
- **회귀 테스트**: 생성 테스트 파일, 기존 테스트 결과
- **교훈**: 같은 유형 버그 예방을 위한 제안

### 7c. 보드 업데이트

`.crew/config.md`에서 `last_task_id`를 읽고 증가시킵니다.

`.crew/board.md`의 `## Done` 섹션에 태스크를 추가합니다:

```markdown
### TASK-[NNN]: 🐛 [버그 요약]
- **Role**: Developer
- **Priority**: P0
- **Depends on**: none
- **Feature**: fix-[slug]
- **Files**: [수정된 파일 목록]
- **Description**: [근본 원인 한 줄 요약]
- **Resolution**: [수정 내용 한 줄 요약]
```

`> Last updated:` 타임스탬프 업데이트.

### 7d. 최종 요약

최종 요약을 제시합니다: 버그 증상, 근본 원인 (유형/위치/설명), 영향 분석 (위험 등급/영향 파일 수/버그 의존 코드), 수정 내용 (파일/검증 결과), 테스트 (회귀 테스트/기존 테스트), 리포트 경로, 태스크 ID.
